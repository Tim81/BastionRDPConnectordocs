Azure Bastion RDP Connector

Ligue-se de forma segura a VM Azure e endereços IP através do Azure Bastion com o cliente Windows RDP nativo.

🗺️ Visão geral

Azure Bastion RDP Connector é uma aplicação de ambiente de trabalho do Windows que facilita o estabelecimento de ligações RDP seguras a máquinas virtuais e endereços IP alojados no Microsoft Azure, encaminhadas através do Azure Bastion — sem expor endereços IP públicos nem abrir portas RDP para a Internet.

Internamente utiliza as suas credenciais de conta Azure e o serviço Azure Bastion. Nunca precisa de gerir certificados, VPNs ou regras de firewall. A aplicação trata de toda a complexidade e entrega o controlo diretamente ao seu cliente padrão de Ambiente de Trabalho Remoto do Windows.

Azure CLI Bastion extension — O modo de túnel também requer a azure-bastion extensão do Azure CLI (separada da própria CLI). Se não estiver instalada, a aplicação detetará isso e oferecerá instalá-la automaticamente antes do primeiro túnel ser criado.

ℹ️

Acessibilidade de rede — O Azure Bastion pode alcançar uma VM de destino ou endereço IP desde que exista conectividade de rede entre eles. Isto inclui:

  • Configurações simples: Bastion implementado no mesmo VNet ou num VNet diretamente peerado.
  • Configurações Enterprise / Landing Zone: Bastion implementado numa Connectivity Landing Zone centralizada ligada através de um hub de Azure Virtual WAN (vWAN). Nesta topologia, o Bastion não é visível dentro das zonas de destino spoke individuais no Portal do Azure — mesmo que possa alcançar todas as VNets spoke através do vWAN. Esta aplicação foi especificamente concebida para resolver este problema, dando aos utilizadores uma interface simples para ligar através de um Bastion gerido centralmente, independentemente da zona de destino onde a sua VM se encontra.

Se não tiver a certeza se o seu Bastion pode alcançar um destino específico, contacte a sua equipa de plataforma Azure ou de rede.

🔐 Primeiro início de sessão

Cada vez que a aplicação inicia, verifica se existe uma cache de tokens MSAL válida na sua máquina. O que acontece a seguir depende dessa verificação:

SituaçãoO que a aplicação faz
Nenhum token em cache encontradoA janela de início de sessão do Microsoft Azure abre imediatamente para que possa iniciar sessão.
Token válido em cache encontradoA aplicação inicia sessão silenciosamente — não aparece janela de início de sessão. Vai diretamente para a seleção de inquilino (se multi-inquilino).
Token expirado / política de AC exige reautenticaçãoA janela de início de sessão abre novamente para satisfazer os requisitos de Acesso Condicional da sua organização (p. ex., MFA diária).
ℹ️

A aplicação armazena o estado de autenticação em quatro ficheiros em %USERPROFILE%\.azure\ — a mesma pasta utilizada pelo Azure CLI:

FicheiroConteúdo
msal_token_cache.binTokens de acesso e atualização MSAL encriptados
azureProfile.jsonLista de subscrições e informações do utilizador com sessão iniciada
clouds.configConfiguração de endpoints de cloud Azure
msal_http_cache.binCache de resposta HTTP

Como estes ficheiros são partilhados com o Azure CLI, a aplicação pode reutilizar uma sessão az login existente e vice-versa.

💡

Para repor a cache de tokens (forçar um novo início de sessão no próximo arranque), elimine os quatro ficheiros de %USERPROFILE%\.azure\:
msal_token_cache.bin, azureProfile.json, clouds.config, msal_http_cache.bin

Em alternativa, utilize o botão Terminar sessão — elimina automaticamente os quatro ficheiros e limpa a cache MSAL num único passo. Note que isto também termina quaisquer sessões ativas do Azure CLI.

Iniciar sessão (sem token em cache)

Quando não há token em cache, a janela de início de sessão do Microsoft Azure abre automaticamente:

1
Selecionar "Conta profissional ou escolar"

Nas opções de início de sessão, escolha a sua conta Microsoft organizacional — não uma conta Microsoft pessoal.

2
Introduzir o seu UPN

Escreva o seu endereço de e-mail de trabalho (Nome Principal de Utilizador) ou selecione-o da lista se tiver sido anteriormente colocado em cache pelo Windows.

3
Concluir MFA se necessário

Siga as instruções no ecrã para qualquer autenticação multifator que a sua organização exija.

Seleção de inquilino

Após um início de sessão bem-sucedido — ou quando a aplicação reinicia e reutiliza um token em cache — aparece uma caixa de diálogo de seleção de inquilino se a sua conta tiver acesso a mais de um inquilino Entra ID (por exemplo, via Azure Lighthouse).

Selecione o inquilino cujos recursos pretende utilizar e clique em OK. Se o inquilino selecionado exigir MFA e o seu token em cache não a satisfizer, aparecerá um aviso de autenticação adicional.

ℹ️

Pode mudar de inquilino a qualquer momento enquanto a aplicação está a correr — as suas definições (última subscrição, Bastion, VM, etc.) são guardadas separadamente por inquilino e restauradas automaticamente quando muda de volta.

Selecionar uma subscrição

Após a seleção do inquilino, a janela principal é carregada. A barra superior mostra a subscrição de Bastion ativa. Se a sua conta tiver acesso a várias subscrições e a pré-selecionada não contiver o Azure Bastion de que necessita, clique em Alterar para abrir o seletor de subscrições.

💡

A subscrição de Bastion (barra superior) é a subscrição que contém o seu recurso Bastion. A subscrição de VM (separador Azure VM) pode ser uma subscrição diferente — as ligações entre subscrições são totalmente suportadas.

O seu token de acesso é guardado em cache de forma segura no disco (cache de tokens MSAL). Nos arranques subsequentes, a aplicação autentica-se silenciosamente — não lhe será pedido que inicie sessão novamente, a não ser que a sua sessão expire ou a política de Acesso Condicional da sua organização exija reautenticação.

🔴

Para terminar sessão e limpar o token em cache, clique no botão vermelho Terminar sessão na barra superior. Aparecerá uma caixa de diálogo de confirmação antes de a aplicação fechar e remover todas as credenciais em cache. O próximo arranque exigirá um início de sessão completo.

🚇 Modo de túnel

O modo de túnel utiliza o comando Azure CLI az network bastion tunnel para criar um encaminhamento de porta local. O tráfego de uma porta da sua máquina (Porta local) é encaminhado através do Azure Bastion para o destino (Porta de destino) na máquina remota.

Como funciona

1
A aplicação verifica os pré-requisitos

A aplicação verifica se o Azure CLI está instalado e se a extensão CLI azure-bastion está presente. Se a extensão estiver em falta, oferece instalá-la automaticamente.

2
A janela minimiza para a bandeja e o túnel abre

A janela principal minimiza automaticamente para não atrapalhar. A aplicação invoca o Azure CLI em segundo plano para estabelecer o túnel de encaminhamento de porta local.

3
O cliente RDP é iniciado automaticamente

O cliente Windows RDP é lançado apontando para localhost:[Porta local]. O tráfego é encaminhado de forma transparente através do Bastion.

4
O túnel permanece aberto

O processo de túnel é executado em segundo plano. Pode vê-lo no separador Túneis ativos e no menu de contexto do ícone da bandeja. Persiste até o parar ou fechar a aplicação.

ℹ️

O modo de túnel é o único método disponível no separador Endereço IP. Suporta múltiplos túneis simultâneos para diferentes destinos.

🏛️ Modo RD Gateway

O modo RD Gateway utiliza a mesma API que o botão Ligar do portal do Azure. A aplicação chama o endpoint Azure Bastion RD Gateway e descarrega um ficheiro .rdp pré-configurado. O cliente Windows RDP liga-se então diretamente ao Azure Bastion como RD Gateway — não é necessário reencaminhamento de porta local.

Como funciona

1
A janela minimiza e o ficheiro RDP é obtido

A janela principal minimiza automaticamente. A aplicação chama a API do Bastion para gerar um ficheiro RDP temporário assinado e guarda-o em segurança numa pasta temporária privada com permissões restritas.

2
O cliente RDP é iniciado automaticamente

mstsc.exe é lançado com o ficheiro RDP descarregado. Liga-se diretamente ao endpoint RD Gateway do Bastion — não é necessário reencaminhamento de porta local.

3
O ficheiro temporário é limpo ao fechar

O ficheiro RDP é mantido apenas durante a sessão da aplicação e é eliminado em segurança quando a aplicação fecha.

💡

O modo RD Gateway não requer Azure CLI. Se a sua organização não permite a instalação do Azure CLI, utilize o modo RD Gateway.

ℹ️

Mecanismo de contingência Entra ID — If Use Entra ID Authentication está ativada mas a API do Bastion a rejeita (por exemplo, num cenário entre inquilinos), a aplicação tenta automaticamente novamente usando autenticação RDP tradicional para que a ligação ainda possa ter sucesso.

Em geral, utilize RD Gateway para o acesso diário a VMs do Azure. Utilize Tunnel quando o Azure CLI já faz parte do seu fluxo de trabalho, ou quando precisa de ligar-se a um endereço IP específico que não é uma VM do Azure — por exemplo:

  • Um servidor ou estação de trabalho no local acessível a partir do Azure Bastion via uma VPN site a site ou circuito Azure ExpressRoute.
  • Um anfitrião noutro fornecedor de cloud (p. ex. AWS, GCP) ligado ao Azure via VPN.
  • Qualquer outro endereço IP roteável de forma privada que o Bastion possa alcançar mas que não tenha registo de VM Azure na sua subscrição.

Nestes casos, o RD Gateway não está disponível — o modo de túnel com um endereço IP direto é a única opção.

⚡ Gestão de energia da VM

Quando uma VM é selecionada no separador Azure VM, a aplicação mostra o seu estado de energia atual e permite iniciá-la se estiver parada.

Indicadores de estado de energia

Em execução — A VM está ligada e pronta para aceitar ligações.
Parada / Desalocada — A VM está desligada. Aparece um botão Iniciar VM.
A iniciar / A parar / Outro — A VM está num estado de transição. Um indicador de progresso é mostrado quando a aplicação inicia um arranque.

Iniciar uma VM parada

1
Selecionar a VM parada

O indicador de estado de energia vermelho e o botão Iniciar VM aparecem.

2
Clicar em Iniciar VM

O botão é substituído por um indicador de progresso (⏳ Iniciando VM…). A aplicação consulta o Azure para obter o estado de energia atualizado.

3
Aguardar o estado Em execução

Quando o indicador fica verde, a VM está pronta. Pode então clicar em Ligar.

💡

O estado Em execução significa que o Azure ligou a VM — mas o sistema operativo convidado ainda precisa de um ou dois minutos para arrancar completamente. Se a sua tentativa de ligação for recusada imediatamente após a VM atingir o estado Em execução, aguarde 2–3 minutos e tente novamente.

⚠️

Iniciar uma VM requer pelo menos a função Virtual Machine Contributor nessa VM. Se o botão Iniciar VM não aparecer, a sua conta pode não ter a permissão necessária.

Assim que um arranque de VM é iniciado, a aplicação consulta o Azure a cada 5 segundos durante até 5 minutos à espera do estado Em execução. Se a VM não atingir esse estado, clique no botão 🔄 Atualizar para verificar o estado atual manualmente.

ℹ️

O botão Ligar no separador Azure VM está desativado sempre que a VM não estiver no estado Em execução. Mesmo que clique em Ligar imediatamente após o indicador de estado ficar verde, a aplicação revalida o estado de energia nesse momento antes de prosseguir.

🏢 Suporte multi-inquilino

Se a sua conta Azure tiver acesso a múltiplos inquilinos Entra ID (por exemplo, via Azure Lighthouse para cenários de fornecedores de serviços geridos), a aplicação suporta a mudança entre inquilinos.

Como funciona a seleção de inquilino

Depois de iniciar sessão, se forem detetados múltiplos inquilinos, ser-lhe-á apresentada uma caixa de diálogo de seleção de inquilino. O inquilino selecionado é exibido junto ao nome da sua conta na barra superior.

Definições por inquilino

As seguintes definições são guardadas separadamente para cada inquilino, pelo que mudar de inquilino restaura o seu contexto anterior para esse inquilino:

Última subscrição Último Bastion Última subscrição VM Última VM Último endereço IP Portas de destino e local Método de ligação Modo de monitor Entra ID Auth setting
💡

A preferência de idioma é global — aplica-se a todos os inquilinos e não é reposta ao mudar.

Se ainda não foi guardado nenhum idioma, a aplicação tenta detetar a sua preferência por esta ordem:

  1. Idioma da interface Windows (CultureInfo.CurrentUICulture)
  2. Idioma de interface Windows instalado
  3. Formato regional do Windows
  4. Definições de idioma do Registo do Windows
  5. Reverte para o inglês

🔧 Resolução de problemas

A aplicação não inicia / a janela não aparece

A aplicação aplica uma política de instância única. Se já estiver a correr na bandeja do sistema, iniciá-la novamente não abrirá uma segunda janela — em vez disso, a janela existente é trazida para o primeiro plano. Verifique a bandeja do sistema para o ícone da aplicação.

Erro «Azure CLI não encontrado» (modo de túnel)

O modo de túnel requer que o Azure CLI (az) esteja instalado e acessível no PATH do sistema. Descarregue e instale a partir de aka.ms/installazurecliwindowsx64. Se o Azure CLI estiver em falta quando tentar ligar, a aplicação mostrará este URL e oferecerá abri-lo diretamente. Em seguida, reinicie a aplicação.

Não aparecem subscrições / lista Bastion vazia

Isto geralmente significa que a sua conta não tem as permissões RBAC necessárias em nenhuma subscrição, ou o inquilino selecionado não contém recursos Bastion. Tente:

  • Clique em Atualizar junto ao seletor de Bastion.
  • Utilize Alterar para selecionar uma subscrição diferente.
  • Terminar sessão e voltar a iniciar sessão se a sua sessão Azure pode ter expirado.
  • Contactar o seu administrador Azure para verificar as suas atribuições de funções.

O cliente RDP abre mas a ligação falha — Azure VM (RD Gateway)

Verifique o seguinte:

  • O estado de energia da VM de destino é Em execução e o sistema operativo teve 2–3 minutos para arrancar completamente.
  • O SKU do Bastion é Standard ou Premium (Basic não suporta ligações de cliente nativo).
  • Nenhuma regra de Grupo de Segurança de Rede (NSG) está a bloquear o tráfego de entrada do Bastion para a VM na porta 3389.
  • Existe conectividade de rede entre o seu Bastion e a VM — seja diretamente, via peering de VNet ou via Virtual WAN.

O cliente RDP abre mas a ligação falha — Azure VM (modo de túnel)

Além das verificações acima, verifique que:

  • O Azure CLI está instalado e atualizado (az upgrade).
  • O processo de túnel ainda está em execução — verifique o separador Túneis ativos. Se o túnel parou inesperadamente, tente ligar novamente para o reiniciar.
💡

Se a porta local configurada já estiver em uso, a aplicação seleciona automaticamente a próxima porta disponível e atualiza a visualização — não é necessária intervenção manual.

O cliente RDP abre mas a ligação falha — endereço IP (modo de túnel)

Além das verificações gerais de túnel acima, verifique que:

  • O endereço IP de destino é alcançável a partir do Azure Bastion — não apenas a partir da sua máquina local. Para anfitriões no local, isto requer uma VPN site a site ou circuito ExpressRoute a funcionar. Para anfitriões noutra cloud, deve existir uma ligação VPN ao Azure.
  • A Porta de destino correta está configurada. Embora 3389 seja a porta RDP padrão, anfitriões no local ou não-Azure podem usar uma porta diferente.
  • Uma firewall ou firewall baseada em host na máquina de destino não está a bloquear o RDP de entrada da gama IP da sub-rede do Bastion.
  • A ligação VPN ou ExpressRoute em si está saudável — consulte a sua equipa de rede se tiver dúvidas.

O botão Iniciar VM não aparece

A sua conta requer pelo menos a função Virtual Machine Contributor na VM de destino para a iniciar. Peça ao seu administrador Azure que conceda esta função, ou inicie a VM manualmente a partir do portal do Azure.

A autenticação Entra ID falha

A autenticação Entra ID só funciona quando a sua conta está no mesmo inquilino que a VM de destino. Se ligar entre inquilinos, desmarque a opção Usar autenticação EntraID e use credenciais Windows tradicionais.

Sessão expirada / pedidos de início de sessão repetidos

A sua organização pode ter uma política de Acesso Condicional que exija reautenticação diária ou MFA. Este é o comportamento esperado. Simplesmente complete o pedido de início de sessão quando aparecer.

Repor definições

Elimine o ficheiro de definições para repor as preferências nos seus valores predefinidos:

%APPDATA%\BastionRDPConnector\settings.json

O próximo arranque irá repor todas as preferências guardadas. Não terá a sessão terminada.

Repor início de sessão / cache de tokens

A forma mais fácil é clicar no botão Terminar sessão — isso elimina automaticamente os quatro ficheiros de autenticação e limpa a cache MSAL num único passo. Se precisar de repor sem abrir a aplicação, elimine manualmente estes quatro ficheiros de %USERPROFILE%\.azure\:

msal_token_cache.bin
azureProfile.json
clouds.config
msal_http_cache.bin
⚠️

Estes ficheiros são partilhados com o Azure CLI. Eliminá-los também terminará quaisquer sessões ativas do Azure CLI — os comandos az voltarão a exigir az login.

Reposição completa (definições + início de sessão)

Para repor completamente a aplicação para um estado limpo, elimine o ficheiro de definições e os quatro ficheiros de autenticação:

%APPDATA%\BastionRDPConnector\settings.json
%USERPROFILE%\.azure\msal_token_cache.bin
%USERPROFILE%\.azure\azureProfile.json
%USERPROFILE%\.azure\clouds.config
%USERPROFILE%\.azure\msal_http_cache.bin

Registo de depuração

A aplicação escreve um registo de depuração detalhado que pode ajudar a diagnosticar problemas de ligação ou autenticação:

%LOCALAPPDATA%\BastionRDPConnector\debug.log
⚠️

O ficheiro de registo é limpo sempre que a aplicação inicia. Se precisar de capturar um registo para resolução de problemas, reproduza o problema na mesma sessão — não reinicie a aplicação antecipadamente.
