Azure Bastion RDP Connector

Stellen Sie sichere Verbindungen zu Azure-VMs und IP-Adressen über Azure Bastion mit dem nativen Windows RDP-Client her — über eine einfache, benutzerfreundliche Oberfläche.

🗺️ Übersicht

Azure Bastion RDP Connector ist eine Windows-Desktop-Anwendung, die sichere RDP-Verbindungen zu virtuellen Maschinen und IP-Adressen in Microsoft Azure über Azure Bastion ermöglicht — ohne öffentliche IP-Adressen freizugeben oder RDP-Ports für das Internet zu öffnen.

Intern verwendet sie Ihre Azure-Kontoanmeldedaten und den Azure Bastion-Dienst. Sie müssen keine Zertifikate, VPNs oder Firewall-Regeln verwalten. Die Anwendung übernimmt die gesamte Komplexität und übergibt die Kontrolle direkt an Ihren Standard-Windows-Remote-Desktop-Client.

✅ Voraussetzungen

Stellen Sie sicher, dass Folgendes in Ihrer Umgebung verfügbar ist, bevor Sie die Anwendung verwenden.

ℹ️

Der RD Gateway-Modus benötigt kein Azure CLI. Wenn Sie ausschließlich die RD Gateway-Verbindungsmethode verwenden, können Sie die Azure CLI-Installation vollständig überspringen.

ℹ️

Azure CLI Bastion extension — Der Tunnelmodus erfordert außerdem die azure-bastion Azure CLI-Erweiterung (getrennt von der CLI selbst). Wenn sie nicht installiert ist, erkennt die Anwendung dies und bietet an, sie vor dem Erstellen des ersten Tunnels automatisch zu installieren.

In Azure

ℹ️

Network reachability — Azure Bastion kann eine Ziel-VM oder IP-Adresse erreichen, solange Netzwerkkonnektivität zwischen ihnen besteht. Dies umfasst:

  • Einfache Setups: Bastion im selben VNet oder in einem direkt gepeertem VNet bereitgestellt.
  • Enterprise / Landing Zone-Setups: Bastion in einer zentralen Connectivity Landing Zone bereitgestellt, die über einen Azure Virtual WAN (vWAN)-Hub verbunden ist. In dieser Topologie ist Bastion in einzelnen Spoke-Landing-Zones im Azure-Portal nicht sichtbar — obwohl es alle Spoke-VNets über das vWAN erreichen kann. Diese Anwendung ist speziell dafür konzipiert, dieses Problem zu lösen und Benutzern eine einfache Schnittstelle zur Verbindung über ein zentral verwaltetes Bastion zu bieten, unabhängig davon, in welcher Landing Zone sich ihre VM befindet.

Wenn Sie nicht sicher sind, ob Ihr Bastion ein bestimmtes Ziel erreichen kann, wenden Sie sich an Ihr Azure-Plattform- oder Netzwerkteam.

🔐 Erste Anmeldung

Jedes Mal, wenn die Anwendung startet, prüft sie, ob ein gültiger MSAL-Token-Cache auf Ihrem Rechner vorhanden ist. Was als nächstes passiert, hängt von dieser Prüfung ab:

SituationWas die App tut

Gültiger gecachter Token gefundenDie App meldet sich still an — es wird kein Anmeldefenster angezeigt. Sie gelangen direkt zur Mandantenauswahl (bei mehreren Mandanten).

Die Anwendung speichert den Authentifizierungsstatus in vier Dateien unter %USERPROFILE%\.azure\ — demselben Ordner, der von der Azure CLI verwendet wird:

💡

Um den Token-Cache zurückzusetzen (erzwinge eine neue Anmeldung beim nächsten Start), löschen Sie alle vier Dateien aus %USERPROFILE%\.azure\:
msal_token_cache.bin, azureProfile.json, clouds.config, msal_http_cache.bin

Alternativ können Sie die Schaltfläche Abmelden verwenden — sie löscht automatisch alle vier Dateien und leert den MSAL-Cache in einem Schritt. Beachten Sie, dass dabei auch aktive Azure CLI-Sitzungen abgemeldet werden.

Anmelden (kein gespeichertes Token)

Wenn kein Token gecacht ist, öffnet sich das Microsoft Azure-Anmeldefenster automatisch:

1
„Geschäfts-, Schul- oder Unikonto" auswählen

Wählen Sie in den Anmeldeoptionen Ihr organisatorisches Microsoft-Konto — kein persönliches Microsoft-Konto.

2
UPN eingeben

Geben Sie Ihre geschäftliche E-Mail-Adresse (User Principal Name) ein oder wählen Sie sie aus der Liste aus, wenn sie zuvor von Windows gecacht wurde.

3

Folgen Sie den Anweisungen auf dem Bildschirm für die von Ihrer Organisation vorgeschriebene Multi-Faktor-Authentifizierung.

Mandantenauswahl

Nach einer erfolgreichen Anmeldung — oder wenn die App neu startet und einen gecachten Token wiederverwendet — erscheint ein Mandantenauswahldialog, wenn Ihr Konto Zugriff auf mehr als einen Entra ID-Mandanten hat (zum Beispiel über Azure Lighthouse).

Wählen Sie den Mandanten aus, mit dessen Ressourcen Sie arbeiten möchten, und klicken Sie auf OK. Wenn der ausgewählte Mandant MFA erfordert und Ihr gecachter Token dies nicht erfüllt, wird eine zusätzliche Authentifizierungsaufforderung angezeigt.

ℹ️

Sie können jederzeit zwischen Mandanten wechseln, während die App läuft — Ihre Einstellungen (letzte Subscription, Bastion, VM usw.) werden separat pro Mandant gespeichert und automatisch wiederhergestellt, wenn Sie zurückwechseln.

Subscription auswählen

Nach der Mandantenauswahl wird das Hauptfenster geladen. Die obere Leiste zeigt die aktuell aktive Bastion Subscription. Wenn Ihr Konto Zugriff auf mehrere Subscriptions hat und die vorausgewählte nicht die benötigte Azure Bastion enthält, klicken Sie auf Ändern, um den Subscription-Auswähler zu öffnen.

💡

Die Bastion Subscription (obere Leiste) ist die Subscription, die Ihre Bastion-Ressource enthält. Die VM Subscription (Registerkarte Azure VM) kann eine andere Subscription sein — abonnementübergreifende Verbindungen werden vollständig unterstützt.

Ihr Zugriffstoken wird sicher auf dem Datenträger gecacht (MSAL-Token-Cache). Bei nachfolgenden Starts meldet sich die Anwendung still an — Sie werden nicht erneut zur Anmeldung aufgefordert, es sei denn, Ihre Sitzung läuft ab oder die Conditional Access-Richtlinie Ihrer Organisation erfordert eine erneute Authentifizierung.

🔴

Um sich abzumelden und den Token-Cache zu leeren, klicken Sie auf die rote Schaltfläche Abmelden in der oberen Leiste. Vor dem Schließen der Anwendung erscheint ein Bestätigungsdialog. Beim nächsten Start ist eine vollständige Anmeldung erforderlich.

🚇 Tunnelmodus

Der Tunnelmodus verwendet den Azure CLI-Befehl az network bastion tunnel , um eine lokale Port-Weiterleitung zu erstellen. Datenverkehr von einem Port auf Ihrem Rechner (Lokaler Port) wird über Azure Bastion zum Ziel (Zielport) auf dem Remote-Computer geleitet.

So funktioniert es

1
Anwendung prüft Voraussetzungen

Die App prüft, ob Azure CLI installiert ist und die azure-bastion-CLI-Erweiterung vorhanden ist. Falls die Erweiterung fehlt, bietet sie an, sie automatisch zu installieren.

2
Fenster minimiert sich in die Taskleiste und Tunnel öffnet sich

Das Hauptfenster minimiert sich automatisch, um nicht im Weg zu sein. Die App ruft Azure CLI im Hintergrund auf, um den lokalen Port-Weiterleitungs-Tunnel herzustellen.

3
RDP-Client wird automatisch gestartet

Der Windows RDP-Client wird gestartet und zeigt auf localhost:[Lokaler Port]. Der Datenverkehr wird transparent über Bastion weitergeleitet.

4
Tunnel bleibt geöffnet

Der Tunnelprozess läuft im Hintergrund. Sie können ihn auf der Registerkarte Aktive Tunnel und im Kontextmenü des Taskleistensymbols sehen. Er bleibt aktiv, bis Sie ihn stoppen oder die Anwendung beenden.

ℹ️

Der Tunnelmodus ist die einzige Methode, die auf der Registerkarte IP-Adresse verfügbar ist. Er unterstützt mehrere gleichzeitige Tunnel zu verschiedenen Zielen.

🏛️ RD Gateway-Modus

Der RD Gateway-Modus verwendet dieselbe API wie die Schaltfläche Verbinden im Azure-Portal. Die Anwendung ruft den Azure Bastion RD Gateway-Endpunkt auf und lädt eine vorkonfigurierte .rdp-Datei herunter. Der Windows RDP-Client verbindet sich dann direkt mit Azure Bastion als RD Gateway — keine lokale Port-Weiterleitung erforderlich.

So funktioniert es

1
Fenster minimiert sich in die Taskleiste und RDP-Datei wird abgerufen

Das Hauptfenster minimiert sich automatisch. Die App ruft die Bastion-API auf, um eine signierte, temporäre RDP-Datei zu generieren, und speichert sie sicher in einem privaten Temp-Ordner mit eingeschränkten Berechtigungen.

2
RDP-Client wird automatisch gestartet

mstsc.exe wird mit der heruntergeladenen RDP-Datei gestartet. Es verbindet sich direkt mit dem RD Gateway-Endpunkt von Bastion — keine lokale Port-Weiterleitung erforderlich.

3
Temporäre Datei wird beim Beenden bereinigt

Die RDP-Datei wird nur für die Dauer der Anwendungssitzung aufbewahrt und beim Schließen der Anwendung sicher gelöscht.

💡

Der RD Gateway-Modus benötigt kein Azure CLI. Wenn Ihre Organisation die Azure CLI-Installation nicht erlaubt, verwenden Sie den RD Gateway-Modus.

ℹ️

Entra ID-Authentifizierungs-Fallback — Wenn EntraID-Authentifizierung verwenden aktiviert ist, die Bastion-API dies jedoch ablehnt (zum Beispiel in einem mandantenübergreifenden Szenario), wiederholt die Anwendung automatisch den Versuch mit traditioneller RDP-Authentifizierung, damit die Verbindung dennoch erfolgreich sein kann.

Im Allgemeinen verwenden Sie RD Gateway für den täglichen Azure-VM-Zugriff. Verwenden Sie Tunnel, wenn Azure CLI bereits Teil Ihres Workflows ist oder wenn Sie eine Verbindung zu einer bestimmten IP-Adresse herstellen müssen, die keine Azure-VM ist — zum Beispiel:

  • Ein lokaler Server oder eine Workstation, der/die von Azure Bastion über eine Site-to-Site-VPN- oder Azure ExpressRoute-Verbindung erreichbar ist.
  • Ein Host bei einem anderen Cloud-Anbieter (z. B. AWS, GCP), der über VPN mit Azure verbunden ist.
  • Jede andere privat routbare IP-Adresse, die Bastion erreichen kann, aber keinen Azure VM-Eintrag in Ihrer Subscription hat.

In diesen Fällen ist RD Gateway nicht verfügbar — der Tunnelmodus mit einer direkten IP-Adresse ist die einzige Option.

⚡ VM-Energieverwaltung

Wenn eine VM auf der Registerkarte Azure VM ausgewählt ist, zeigt die Anwendung ihren aktuellen Energiestatus an und ermöglicht es Ihnen, sie zu starten, wenn sie gestoppt ist.

Energiestatus-Indikatoren

Wird ausgeführt — Die VM ist eingeschaltet und bereit, Verbindungen anzunehmen.
Gestoppt / Freigegeben — Die VM ist ausgeschaltet. Eine Schaltfläche VM starten erscheint.
Starten / Stoppen / Sonstiges — Die VM befindet sich in einem Übergangszustand. Ein Fortschrittsindikator wird angezeigt, wenn die App einen Start einleitet.

Gestoppte VM starten

1
Gestoppte VM auswählen

Der rote Energiestatus-Indikator und die Schaltfläche VM starten erscheinen.

2
Auf „VM starten" klicken

Die Schaltfläche wird durch einen Fortschrittsindikator (⏳ VM wird gestartet…) ersetzt. Die Anwendung fragt Azure nach dem aktualisierten Energiestatus ab.

3
Auf den Status „Wird ausgeführt" warten

Wenn der Indikator grün wird, ist die VM bereit. Sie können dann auf Verbinden klicken.

💡

Der Status Wird ausgeführt bedeutet, dass Azure die VM eingeschaltet hat — aber das Gastbetriebssystem benötigt noch ein bis zwei Minuten zum vollständigen Starten. Wenn Ihr Verbindungsversuch unmittelbar nach Erreichen des Status abgelehnt wird, warten Sie 2–3 Minuten und versuchen Sie es erneut.

⚠️

Das Starten einer VM erfordert mindestens die Rolle Virtual Machine Contributor für diese VM. Wenn die Schaltfläche „VM starten" nicht angezeigt wird, fehlen Ihrem Konto möglicherweise die erforderlichen Berechtigungen.

Sobald ein VM-Start eingeleitet wurde, fragt die Anwendung Azure alle 5 Sekunden für bis zu 5 Minuten auf den Status „Wird ausgeführt" ab. Wenn die VM diesen Status nicht erreicht, klicken Sie auf die Schaltfläche 🔄 Aktualisieren, um den aktuellen Status manuell zu prüfen.

ℹ️

Die Schaltfläche Verbinden auf der Registerkarte Azure VM ist deaktiviert, wenn sich die VM nicht im Status „Wird ausgeführt" befindet. Selbst wenn Sie sofort nach dem Grünwerden des Statusindikators auf Verbinden klicken, überprüft die Anwendung den Energiestatus zu diesem Zeitpunkt erneut.

🏢 Multi-Mandanten-Unterstützung

Wenn Ihr Azure-Konto Zugriff auf mehrere Entra ID-Mandanten hat (zum Beispiel über Azure Lighthouse für Managed Service Provider-Szenarien), unterstützt die Anwendung das Wechseln zwischen Mandanten.

So funktioniert die Mandantenauswahl

Nach dem Anmelden wird Ihnen, wenn mehrere Mandanten erkannt werden, ein Mandantenauswahldialog angezeigt. Der ausgewählte Mandant wird neben Ihrem Kontonamen in der oberen Leiste angezeigt.

🔧 Fehlerbehebung

Die Anwendung startet nicht / Fenster erscheint nicht

Die Anwendung erzwingt eine Einzelinstanz-Richtlinie. Wenn sie bereits im Benachrichtigungsbereich läuft, öffnet ein erneutes Starten kein zweites Fenster — stattdessen wird das vorhandene Fenster in den Vordergrund gebracht. Prüfen Sie den Benachrichtigungsbereich auf das Anwendungssymbol.

Fehler „Azure CLI nicht gefunden" (Tunnelmodus)

Der Tunnelmodus erfordert, dass Azure CLI (az) installiert und über den System-PATH zugänglich ist. Herunterladen und installieren von aka.ms/installazurecliwindowsx64. Wenn Azure CLI fehlt, wenn Sie versuchen, eine Verbindung herzustellen, zeigt die Anwendung diese URL an und bietet an, sie direkt zu öffnen. Starten Sie dann die Anwendung neu.

Keine Subscriptions sichtbar / Bastion-Liste ist leer

Dies bedeutet normalerweise, dass Ihr Konto nicht über die erforderlichen RBAC-Berechtigungen für eine Subscription verfügt oder der ausgewählte Mandant keine Bastion-Ressourcen enthält. Versuchen Sie:

  • Klicken Sie auf Aktualisieren neben dem Bastion-Selektor.
  • Verwenden Sie Ändern, um eine andere Subscription auszuwählen.
  • Ab- und wieder anmelden, wenn Ihre Azure-Sitzung möglicherweise abgelaufen ist.
  • Kontaktieren Sie Ihren Azure-Administrator, um Ihre Rollenzuweisungen zu überprüfen.

RDP-Client öffnet, Verbindung schlägt fehl — Azure VM (RD Gateway)

Stellen Sie Folgendes sicher:

  • Der Energiestatus der Ziel-VM ist Wird ausgeführt und das Betriebssystem hatte 2–3 Minuten zum vollständigen Hochfahren.
  • Die Bastion SKU ist Standard oder Premium (Basic unterstützt keine nativen Client-Verbindungen).
  • Keine Network Security Group (NSG)-Regel blockiert eingehenden Datenverkehr von Bastion zur VM auf Port 3389.
  • Netzwerkkonnektivität zwischen Ihrem Bastion und der VM besteht — direkt, über VNet-Peering oder über Virtual WAN.

RDP-Client öffnet, Verbindung schlägt fehl — Azure VM (Tunnelmodus)

Überprüfen Sie zusätzlich zu den obigen Prüfungen Folgendes:

  • Azure CLI ist installiert und aktuell (az upgrade).
  • Der Tunnelprozess läuft noch — prüfen Sie die Registerkarte Aktive Tunnel. Wenn der Tunnel unerwartet gestoppt hat, versuchen Sie, erneut zu verbinden.
💡

Wenn der konfigurierte lokale Port bereits belegt ist, wählt die Anwendung automatisch den nächsten verfügbaren Port und aktualisiert die Anzeige — kein manueller Eingriff erforderlich.

RDP-Client öffnet, Verbindung schlägt fehl — IP-Adresse (Tunnelmodus)

Überprüfen Sie zusätzlich zu den allgemeinen Tunnel-Prüfungen oben Folgendes:

  • Die Ziel-IP-Adresse ist von Azure Bastion aus erreichbar — nicht nur von Ihrem lokalen Rechner. Für lokale Hosts erfordert dies eine funktionierende Site-to-Site-VPN oder ExpressRoute-Verbindung. Für Hosts in einer anderen Cloud muss eine VPN-Verbindung zu Azure bestehen.
  • Der richtige Zielport ist eingestellt. Obwohl 3389 der Standard-RDP-Port ist, können lokale oder Nicht-Azure-Hosts einen anderen Port verwenden.
  • Eine Firewall oder hostbasierte Firewall auf dem Zielrechner blockiert kein eingehendes RDP aus dem IP-Bereich des Bastion-Subnetzes.
  • Die VPN- oder ExpressRoute-Verbindung selbst ist gesund — wenden Sie sich an Ihr Netzwerkteam, wenn Sie unsicher sind.

Schaltfläche VM starten erscheint nicht

Ihr Konto benötigt mindestens die Rolle Virtual Machine Contributor für die Ziel-VM, um sie zu starten. Bitten Sie Ihren Azure-Administrator, diese Rolle zu erteilen, oder starten Sie die VM manuell über das Azure-Portal.

Entra ID-Authentifizierung schlägt fehl

Die Entra ID-Authentifizierung funktioniert nur, wenn sich Ihr Konto im selben Mandanten wie die Ziel-VM befindet. Wenn Sie mandantenübergreifend verbinden, deaktivieren Sie die Option EntraID-Authentifizierung verwenden und verwenden Sie stattdessen traditionelle Windows-Anmeldeinformationen.

Sitzung abgelaufen / wiederholte Anmeldeaufforderungen

Ihre Organisation hat möglicherweise eine Conditional Access-Richtlinie, die eine tägliche erneute Authentifizierung oder MFA erfordert. Dies ist das erwartete Verhalten. Schließen Sie einfach die Anmeldeaufforderung ab, wenn sie erscheint.

Einstellungen zurücksetzen

Löschen Sie die Einstellungsdatei, um die Einstellungen auf die Standardwerte zurückzusetzen:

%APPDATA%\BastionRDPConnector\settings.json

Beim nächsten Start werden alle gespeicherten Einstellungen zurückgesetzt. Sie werden nicht abgemeldet.

Anmeldung / Token-Cache zurücksetzen

Am einfachsten klicken Sie auf die Schaltfläche Abmelden — das löscht automatisch alle vier Authentifizierungsdateien und leert den MSAL-Cache in einem Schritt. Wenn Sie ohne Öffnen der Anwendung zurücksetzen müssen, löschen Sie diese vier Dateien manuell aus %USERPROFILE%\.azure\:

msal_token_cache.bin
azureProfile.json
clouds.config
msal_http_cache.bin
⚠️

Diese Dateien werden mit der Azure CLI geteilt. Das Löschen meldet auch aktive Azure CLI-Sitzungen ab — az-Befehle erfordern dann erneut az login.

Vollständiges Zurücksetzen (Einstellungen + Anmeldung)

Um die Anwendung vollständig auf einen sauberen Zustand zurückzusetzen, löschen Sie die Einstellungsdatei und alle vier Authentifizierungsdateien:

%APPDATA%\BastionRDPConnector\settings.json
%USERPROFILE%\.azure\msal_token_cache.bin
%USERPROFILE%\.azure\azureProfile.json
%USERPROFILE%\.azure\clouds.config
%USERPROFILE%\.azure\msal_http_cache.bin

Debug-Protokoll

Die Anwendung schreibt ein detailliertes Debug-Protokoll, das bei der Diagnose von Verbindungs- oder Authentifizierungsproblemen helfen kann:

%LOCALAPPDATA%\BastionRDPConnector\debug.log
⚠️

Die Protokolldatei wird bei jedem Start der Anwendung gelöscht. Wenn Sie ein Protokoll zur Fehlerbehebung erfassen müssen, reproduzieren Sie das Problem in derselben Sitzung — starten Sie die Anwendung vorher nicht neu.
