Azure Bastion RDP Connector

Connectez-vous en toute sécurité aux VM Azure et aux adresses IP via Azure Bastion avec le client Windows RDP natif — via une interface simple et conviviale.

🗺️ Aperçu

Azure Bastion RDP Connector est une application de bureau Windows qui facilite l'établissement de connexions RDP sécurisées vers des machines virtuelles et des adresses IP hébergées dans Microsoft Azure, acheminées via Azure Bastion — sans exposer d'adresses IP publiques ni ouvrir de ports RDP sur Internet.

En coulisses, elle utilise vos informations d'identification Azure et le service Azure Bastion. Vous n'avez jamais besoin de gérer des certificats, des VPN ou des règles de pare-feu. L'application gère toute la complexité et remet le contrôle directement à votre client Windows Remote Desktop standard.

ℹ️

Accessibilité réseau — Azure Bastion peut atteindre une VM cible ou une adresse IP tant que la connectivité réseau existe entre eux. Cela inclut :

  • Configurations simples : Bastion déployé dans le même VNet ou dans un VNet directement peeré.
  • Configurations Enterprise / Landing Zone : Bastion déployé dans une Connectivity Landing Zone centralisée connectée via un hub Azure Virtual WAN (vWAN). Dans cette topologie, Bastion n'est pas visible dans les zones d'atterrissage spoke individuelles dans le portail Azure — même s'il peut atteindre tous les VNets spoke via le vWAN. Cette application est spécifiquement conçue pour résoudre ce problème, offrant aux utilisateurs une interface simple pour se connecter via un Bastion géré de manière centralisée, quelle que soit la zone d'atterrissage où se trouve leur VM.

Si vous n'êtes pas sûr que votre Bastion puisse atteindre une cible spécifique, contactez votre équipe Azure Platform ou réseau.

🔐 Première connexion

Chaque fois que l'application démarre, elle vérifie si un cache de jetons MSAL valide existe sur votre machine. Ce qui se passe ensuite dépend de cette vérification :

SituationCe que fait l'application
Aucun jeton mis en cacheLa fenêtre de connexion Microsoft Azure s'ouvre immédiatement pour vous permettre de vous connecter.
Jeton mis en cache valide trouvéL'application se connecte silencieusement — aucune fenêtre de connexion n'apparaît. Vous passez directement à la sélection du locataire (si multi-locataire).
Jeton expiré / la stratégie AC exige une réauthentificationLa fenêtre de connexion s'ouvre à nouveau pour satisfaire aux exigences d'accès conditionnel de votre organisation (par ex. MFA quotidien).
ℹ️

L'application stocke l'état d'authentification dans quatre fichiers dans %USERPROFILE%\.azure\ — le même dossier utilisé par Azure CLI :

FichierContenu
msal_token_cache.binJetons d'accès et de rafraîchissement MSAL chiffrés
azureProfile.jsonListe des abonnements et informations de l'utilisateur connecté
clouds.configConfiguration des points de terminaison Azure cloud
msal_http_cache.binCache de réponse HTTP

Ces fichiers étant partagés avec Azure CLI, l'application peut réutiliser une session az login existante et vice versa.

💡

Pour réinitialiser le cache des jetons (forcer une nouvelle connexion au prochain démarrage), supprimez les quatre fichiers de %USERPROFILE%\.azure\ :
msal_token_cache.bin, azureProfile.json, clouds.config, msal_http_cache.bin

Vous pouvez également utiliser le bouton Se déconnecter — il supprime automatiquement les quatre fichiers et vide le cache MSAL en une seule étape. Notez que cela déconnecte également toutes les sessions Azure CLI actives.

Connexion (aucun jeton en cache)

Lorsqu'aucun jeton n'est mis en cache, la fenêtre de connexion Microsoft Azure s'ouvre automatiquement :

1
Sélectionner « Compte professionnel ou scolaire »

Dans les options de connexion, choisissez votre compte Microsoft organisationnel — pas un compte Microsoft personnel.

2
Saisir votre UPN

Saisissez votre adresse e-mail professionnelle (nom d'utilisateur principal) ou sélectionnez-la dans la liste si elle a déjà été mise en cache par Windows.

3
Effectuer la MFA si nécessaire

Suivez les instructions à l'écran pour toute authentification multi-facteurs que votre organisation impose.

Sélection du locataire

Après une connexion réussie — ou lorsque l'application redémarre et réutilise un jeton mis en cache — une boîte de dialogue de sélection du locataire apparaît si votre compte a accès à plus d'un locataire Entra ID (par exemple via Azure Lighthouse).

Sélectionnez le locataire dont vous souhaitez utiliser les ressources, puis cliquez sur OK. Si le locataire sélectionné exige la MFA et que votre jeton mis en cache ne la satisfait pas, une invite d'authentification supplémentaire apparaîtra.

ℹ️

Vous pouvez changer de locataire à tout moment pendant que l'application est en cours d'exécution — vos paramètres (dernier abonnement, Bastion, VM, etc.) sont stockés séparément par locataire et restaurés automatiquement lorsque vous revenez.

Sélectionner un abonnement

Après la sélection du locataire, la fenêtre principale se charge. La barre supérieure affiche l'abonnement Bastion actif. Si votre compte a accès à plusieurs abonnements et que celui présélectionné ne contient pas le Azure Bastion dont vous avez besoin, cliquez sur Modifier pour ouvrir le sélecteur d'abonnements.

💡

L'abonnement Bastion (barre supérieure) est l'abonnement qui contient votre ressource Bastion. L'abonnement VM (onglet Azure VM) peut être un abonnement différent — les connexions inter-abonnements sont entièrement prises en charge.

Votre jeton d'accès est mis en cache de manière sécurisée sur le disque (cache de jetons MSAL). Lors des démarrages suivants, l'application s'authentifie silencieusement — vous ne serez pas invité à vous reconnecter sauf si votre session expire ou si la politique d'accès conditionnel de votre organisation exige une nouvelle authentification.

🔴

Pour vous déconnecter et effacer le jeton mis en cache, cliquez sur le bouton rouge Se déconnecter dans la barre supérieure. Une boîte de dialogue de confirmation apparaîtra avant que l'application ne se ferme et ne supprime toutes les informations d'identification mises en cache. Le prochain démarrage nécessitera une connexion complète.

🚇 Mode tunnel

Le mode tunnel utilise la commande Azure CLI az network bastion tunnel pour créer un transfert de port local. Le trafic d'un port de votre machine (Port local) est acheminé via Azure Bastion vers la cible (Port cible) sur la machine distante.

Comment ça fonctionne

1
L'application vérifie les prérequis

L'application vérifie qu'Azure CLI est installé et que l'extension CLI azure-bastion est présente. Si l'extension est manquante, elle propose de l'installer automatiquement.

2
La fenêtre se réduit dans la barre des tâches et le tunnel s'ouvre

La fenêtre principale se minimise automatiquement pour ne pas gêner. L'application invoque Azure CLI en arrière-plan pour établir le tunnel de transfert de port local.

3
Le client RDP se lance automatiquement

Le client Windows RDP est lancé en pointant vers localhost:[Port local]. Le trafic est transféré de manière transparente via Bastion.

4
Le tunnel reste ouvert

Le processus de tunnel s'exécute en arrière-plan. Vous pouvez le voir dans l'onglet Tunnels actifs et dans le menu contextuel de l'icône de la barre des tâches. Il persiste jusqu'à ce que vous l'arrêtiez ou fermiez l'application.

ℹ️

Le mode tunnel est la seule méthode disponible dans l'onglet Adresse IP. Il prend en charge plusieurs tunnels simultanés vers différentes cibles.

🏛️ Mode RD Gateway

Le mode RD Gateway utilise la même API que le bouton Connecter du portail Azure. L'application appelle le point de terminaison Azure Bastion RD Gateway et télécharge un fichier .rdp préconfiguré. Le client Windows RDP se connecte ensuite directement à Azure Bastion en tant que RD Gateway — aucune redirection de port locale n'est nécessaire.

Comment ça fonctionne

1
La fenêtre se réduit et le fichier RDP est récupéré

La fenêtre principale se minimise automatiquement. L'application appelle l'API Bastion pour générer un fichier RDP temporaire signé et le sauvegarde de manière sécurisée dans un dossier temporaire privé avec des autorisations restreintes.

2
Le client RDP se lance automatiquement

mstsc.exe se lance avec le fichier RDP téléchargé. Il se connecte directement au point de terminaison RD Gateway de Bastion — aucune redirection de port locale n'est nécessaire.

3
Le fichier temporaire est nettoyé à la fermeture

Le fichier RDP n'est conservé que pour la durée de la session de l'application et est supprimé de manière sécurisée lorsque l'application se ferme.

💡

Le mode RD Gateway ne nécessite pas Azure CLI. Si votre organisation n'autorise pas l'installation d'Azure CLI, utilisez le mode RD Gateway.

ℹ️

Mécanisme de secours Entra ID — If Use Entra ID Authentication est activée mais que l'API Bastion la rejette (par exemple dans un scénario inter-locataires), l'application réessaie automatiquement avec l'authentification RDP traditionnelle pour que la connexion puisse quand même réussir.

En général, utilisez RD Gateway pour l'accès quotidien aux VM Azure. Utilisez Tunnel lorsque Azure CLI fait déjà partie de votre flux de travail, ou lorsque vous devez vous connecter à une adresse IP spécifique qui n'est pas une VM Azure — par exemple :

  • Un serveur ou poste de travail sur site accessible depuis Azure Bastion via un VPN site à site ou un circuit Azure ExpressRoute.
  • Un hôte chez un autre fournisseur cloud (p. ex. AWS, GCP) connecté à Azure via VPN.
  • Toute autre adresse IP routable en privé que Bastion peut atteindre mais qui n'a pas d'enregistrement VM Azure dans votre abonnement.

Dans ces cas, RD Gateway n'est pas disponible — le mode tunnel avec une adresse IP directe est la seule option.

⚡ Gestion de l'alimentation VM

Lorsqu'une VM est sélectionnée dans l'onglet Azure VM, l'application affiche son état d'alimentation actuel et vous permet de la démarrer si elle est arrêtée.

Indicateurs d'état d'alimentation

En cours d'exécution — La VM est sous tension et prête à accepter des connexions.
Arrêtée / Désallouée — La VM est éteinte. Un bouton Démarrer la VM apparaît.
Démarrage / Arrêt / Autre — La VM est dans un état de transition. Un indicateur de progression est affiché lorsque l'application lance un démarrage.

Démarrage d'une VM arrêtée

1
Sélectionner la VM arrêtée

L'indicateur d'état d'alimentation rouge et le bouton Démarrer la VM apparaissent.

2
Cliquer sur Démarrer la VM

Le bouton est remplacé par un indicateur de progression (⏳ Démarrage de la VM…). L'application interroge Azure pour connaître l'état d'alimentation mis à jour.

3
Attendre l'état En cours d'exécution

Quand l'indicateur devient vert, la VM est prête. Vous pouvez alors cliquer sur Connecter.

💡

L'état En cours d'exécution signifie qu'Azure a mis la VM sous tension — mais le système d'exploitation invité a encore besoin d'une à deux minutes pour démarrer complètement. Si votre tentative de connexion est refusée immédiatement après que la VM atteint cet état, attendez 2–3 minutes et réessayez.

⚠️

Le démarrage d'une VM nécessite au minimum le rôle Virtual Machine Contributor sur cette VM. Si le bouton Démarrer la VM n'apparaît pas, votre compte manque peut-être de l'autorisation requise.

Une fois le démarrage d'une VM initié, l'application interroge Azure toutes les 5 secondes pendant 5 minutes en attendant l'état En cours d'exécution. Si la VM n'atteint pas cet état, cliquez sur le bouton 🔄 Actualiser pour vérifier manuellement l'état actuel.

ℹ️

Le bouton Connecter sur l'onglet Azure VM est désactivé lorsque la VM n'est pas dans l'état En cours d'exécution. Même si vous cliquez sur Connecter immédiatement après que l'indicateur d'état passe au vert, l'application revalide l'état d'alimentation à ce moment avant de continuer.

🏢 Support multi-locataires

Si votre compte Azure a accès à plusieurs locataires Entra ID (par exemple, via Azure Lighthouse pour des scénarios de fournisseurs de services gérés), l'application prend en charge le changement de locataire.

Fonctionnement de la sélection du locataire

Après la connexion, si plusieurs locataires sont détectés, une boîte de dialogue de sélection du locataire vous sera présentée. Le locataire sélectionné est affiché à côté de votre nom de compte dans la barre supérieure.

Paramètres par locataire

Les paramètres suivants sont stockés séparément pour chaque locataire, de sorte que le changement de locataire restaure votre contexte précédent pour ce locataire :

Dernier abonnement Dernier Bastion Dernier abonnement VM Dernière VM Dernière adresse IP Ports cible et local Méthode de connexion Mode moniteur Entra ID Auth setting
💡

La préférence de langue est globale — elle s'applique à tous les locataires et n'est pas réinitialisée lors d'un changement.

📌 Zone de notification

L'application ajoute une icône dans la zone de notification Windows (barre des tâches). Cela vous permet de la garder en cours d'exécution en arrière-plan tout en gardant votre barre des tâches ordonnée.

ActionRésultat
Double-clic sur l'icône de la zone de notificationRestaure la fenêtre principale au premier plan.
Clic droit sur l'icône de la zone de notificationOuvre un menu contextuel listant tous les tunnels actifs (chacun en sous-menu), un élément Afficher la fenêtre principale et un élément Quitter.
Sous-menu tunnelChaque tunnel actif apparaît comme un sous-menu affichant la cible et le port local. De là, vous pouvez Connecter RDP (relancer le client RDP) ou Arrêter le tunnel sans ouvrir la fenêtre principale.
NotificationsL'icône de la zone de notification affiche des notifications toast pour tunnel démarré, tunnel connecté, tunnel arrêté, VM démarrée et erreurs de connexion.
💡

Fermer la fenêtre principale ou la réduire masque la fenêtre et la supprime de la barre des tâches — l'application continue de s'exécuter dans la zone de notification. Pour quitter complètement, faites un clic droit sur l'icône de la zone de notification et choisissez Quitter.

Les changements de langue prennent effet immédiatement sans redémarrer l'application. Tous les textes de l'interface — libellés, boutons, messages et info-bulles — se mettent à jour en direct. La langue sélectionnée est sauvegardée globalement et persiste entre les démarrages.

Ordre de détection automatique

Si aucune langue n'a encore été sauvegardée, l'application essaie de détecter votre préférence dans cet ordre :

  1. Langue de l'interface Windows (CultureInfo.CurrentUICulture)
  2. Langue d'interface Windows installée
  3. Format régional Windows
  4. Paramètres de langue du Registre Windows
  5. Revient à l'anglais

💾 Paramètres et persistance

L'application sauvegarde et restaure automatiquement vos préférences entre les sessions. Vous n'avez jamais besoin de modifier manuellement un fichier — tout est géré par l'application.

🔧 Dépannage

L'application ne démarre pas / la fenêtre n'apparaît pas

L'application applique une stratégie d'instance unique. Si elle est déjà en cours d'exécution dans la zone de notification, la relancer n'ouvrira pas de seconde fenêtre — la fenêtre existante est mise au premier plan. Vérifiez la zone de notification pour l'icône de l'application.

Erreur « Azure CLI introuvable » (mode tunnel)

Le mode tunnel nécessite qu'Azure CLI (az) soit installé et accessible sur le PATH de votre système. Téléchargez et installez depuis aka.ms/installazurecliwindowsx64. Si Azure CLI est manquant lorsque vous essayez de vous connecter, l'application affiche cette URL et propose de l'ouvrir directement. Redémarrez ensuite l'application.

Aucun abonnement visible / liste Bastion vide

Cela signifie généralement que votre compte n'a pas les autorisations RBAC requises sur un abonnement, ou que le locataire sélectionné ne contient pas de ressources Bastion. Essayez :

  • Cliquez sur Actualiser à côté du sélecteur de Bastion.
  • Utilisez Modifier pour sélectionner un autre abonnement.
  • Se déconnecter et se reconnecter si votre session Azure a peut-être expiré.
  • Contactez votre administrateur Azure pour vérifier vos attributions de rôles.

Le client RDP s'ouvre mais la connexion échoue — Azure VM (RD Gateway)

Vérifiez que :

  • L'état d'alimentation de la VM cible est En cours d'exécution et le système d'exploitation a eu 2–3 minutes pour démarrer complètement.
  • Le SKU Bastion est Standard ou Premium (Basic ne prend pas en charge les connexions client natives).
  • Aucune règle de groupe de sécurité réseau (NSG) ne bloque le trafic entrant de Bastion vers la VM sur le port 3389.
  • La connectivité réseau entre votre Bastion et la VM existe — directement, via le peering VNet ou via Virtual WAN.

Le client RDP s'ouvre mais la connexion échoue — Azure VM (mode tunnel)

En plus des vérifications ci-dessus, vérifiez que :

  • Azure CLI est installé et à jour (az upgrade).
  • Le processus de tunnel est toujours en cours — vérifiez l'onglet Tunnels actifs. Si le tunnel s'est arrêté de façon inattendue, essayez de vous reconnecter pour le redémarrer.
💡

Si le port local configuré est déjà utilisé, l'application sélectionne automatiquement le prochain port disponible et met à jour l'affichage — aucune intervention manuelle n'est nécessaire.

Le client RDP s'ouvre mais la connexion échoue — adresse IP (mode tunnel)

En plus des vérifications générales du tunnel ci-dessus, vérifiez que :

  • L'adresse IP cible est accessible depuis Azure Bastion — pas seulement depuis votre machine locale. Pour les hôtes sur site, cela nécessite un VPN site à site opérationnel ou un circuit ExpressRoute. Pour les hôtes dans un autre cloud, une connexion VPN vers Azure doit être en place.
  • Le Port cible correct est configuré. Bien que 3389 soit le port RDP standard, les hôtes sur site ou non-Azure peuvent utiliser un port différent.
  • Un pare-feu ou un pare-feu basé sur l'hôte sur la machine cible ne bloque pas le RDP entrant depuis la plage IP du sous-réseau Bastion.
  • La connexion VPN ou ExpressRoute elle-même est saine — consultez votre équipe réseau en cas de doute.

Le bouton Démarrer la VM n'apparaît pas

Votre compte nécessite au minimum le rôle Virtual Machine Contributor sur la VM cible pour la démarrer. Demandez à votre administrateur Azure d'accorder ce rôle, ou démarrez la VM manuellement depuis le portail Azure.

L'authentification Entra ID échoue

L'authentification Entra ID ne fonctionne que lorsque votre compte est dans le même locataire que la VM cible. Si vous vous connectez entre locataires, décochez l'option Utiliser l'authentification EntraID et utilisez à la place les informations d'identification Windows traditionnelles.

Session expirée / invites de connexion répétées

Votre organisation peut avoir une politique d'accès conditionnel exigeant une réauthentification quotidienne ou la MFA. C'est un comportement attendu. Complétez simplement l'invite de connexion lorsqu'elle apparaît.

Réinitialisation des paramètres

Supprimez le fichier de paramètres pour réinitialiser les préférences à leurs valeurs par défaut :

%APPDATA%\BastionRDPConnector\settings.json

Le prochain démarrage réinitialisera toutes les préférences sauvegardées. Vous ne serez pas déconnecté.

Réinitialisation du cache de connexion / jetons

Le moyen le plus simple est de cliquer sur le bouton Se déconnecter — cela supprime automatiquement les quatre fichiers d'authentification et vide le cache MSAL en une seule étape. Si vous devez réinitialiser sans ouvrir l'application, supprimez manuellement ces quatre fichiers de %USERPROFILE%\.azure\:

msal_token_cache.bin
azureProfile.json
clouds.config
msal_http_cache.bin
⚠️

Ces fichiers sont partagés avec Azure CLI. Les supprimer déconnectera également les sessions Azure CLI actives — les commandes az nécessiteront à nouveau az login.

Réinitialisation complète (paramètres + connexion)

Pour réinitialiser complètement l'application à un état propre, supprimez le fichier de paramètres et les quatre fichiers d'authentification :

%APPDATA%\BastionRDPConnector\settings.json
%USERPROFILE%\.azure\msal_token_cache.bin
%USERPROFILE%\.azure\azureProfile.json
%USERPROFILE%\.azure\clouds.config
%USERPROFILE%\.azure\msal_http_cache.bin

Journal de débogage

L'application écrit un journal de débogage détaillé qui peut aider à diagnostiquer les problèmes de connexion ou d'authentification :

%LOCALAPPDATA%\BastionRDPConnector\debug.log
⚠️

Le fichier journal est effacé à chaque démarrage de l'application. Si vous devez capturer un journal pour le dépannage, reproduisez le problème dans la même session — ne redémarrez pas l'application au préalable.
